UJENZIXFORM — WORK STILL PENDING OR INCOMPLETE (PLAIN-LANGUAGE SUMMARY)
================================================================================

This note lists areas of the application that are partly done, marked as future
work in code, or still thin compared to a full production product. It is written
in paragraph form so you can open it in Notepad or Word, share it, or print or
save as PDF from your editor or printer dialog.

The list reflects the codebase and product discussions as of early 2026. It is
not a promise of delivery order; use it as a planning checklist.


DIFFERENTIATORS — WHAT EXISTS NOW VS WHAT COMES NEXT
--------------------------------------------------------------------------------

Site vision / ML — The foundation is in the repo and deployable: table
site_vision_events, Python worker (motion + optional generic YOLO), and
packaged ops via workers/site-vision/Dockerfile and docker-compose.yml (Linux,
mini PC, WSL2, or cloud VM). “Reliable ops” for pilots means: run that container
(or the Python venv) on site, point cameras.stream_url at RTSP/HLS OpenCV can
open, and confirm events on Analytics → Site vision. Custom CV (per-material
models, PPE compliance products) is a later phase after this path is stable in
the field.

PTZ / two-way audio — The app stores supports_ptz and supports_two_way_audio and
shows monitoring UI scaffolding. Actually moving the lens or browser talk-back
still needs a secure gateway (Supabase Edge Function, vendor cloud API, ONVIF
relay, or WebRTC server): credentials and control must not live in the browser.
Ship that gateway only after core streaming and site vision ops are stable;
until then the UI is intentionally preview-only.


SITE VISION AND MACHINE LEARNING
--------------------------------------------------------------------------------

The site vision pipeline is a working foundation (not a blank slate): a database
table for events, a Python worker that can read camera streams, motion
detection, and optional YOLO for generic object classes. What is not yet
delivered as a finished product is custom-trained computer vision for every
construction material label, and any certified workplace PPE compliance
product. Those would require more data, model work, or a third-party integration.

The database includes a field for image references, but the worker does not yet
fully attach saved thumbnail images to every alert in a polished way. Events
today are mostly metadata plus optional payloads.

Writing events with the Supabase service role on a laptop is acceptable for a
pilot. For field PCs without a service role, the repo ships an optional Edge
path (signed Storage upload ticket) documented in docs/SITE_VISION_ADVANCED_INTEGRATION.md.

Consumer-style links such as GoPro cloud gallery pages are still not valid
stream addresses for the worker. The product still assumes you supply RTSP, HLS
with a real stream URL, or another format OpenCV and VLC can open directly.


MONITORING AND ACCESS CONTROL
--------------------------------------------------------------------------------

Row-level security for cameras and site_vision_events is tightened in the
consolidated migration 20260331120000_platform_security_consolidated.sql:
project-scoped camera SELECT for authenticated users, admin manage-all, no anon
table reads; site vision events follow the same camera access; guest monitoring
links use a SECURITY DEFINER RPC (resolve_monitoring_access_code) so assigned
cameras load without exposing the cameras table to anon REST.
Optional Edge layers (staff login throttling + stream URL via Function) are
documented in docs/EDGE_THROTTLING_AND_STREAM_RELAY.md (VITE_* flags). Full
signed-ticket or byte-level HLS relay is still future work for the strictest
deployments.

The Analytics site vision screen reads events as an authenticated Supabase user.
Some administrator sessions that rely only on browser local storage without a
normal Supabase JWT may not see the same rows. Aligning admin sign-in with
database rules is still an area to clarify for a consistent experience.


BROADER APPLICATION FEATURES
--------------------------------------------------------------------------------

Central error reporting to a service such as Sentry is referenced in code but
not fully wired; today errors are mostly console and user-facing toasts.

Parts of delivery and tracking logic still have placeholders for SMS and email
notifications when phone or email is available; those channels may not fire in
every path until implemented end to end.

In the admin dashboard, some registration workflows still lack a full “view
details” experience for every row. The professional builder dashboard may defer
re-enabling certain client name fields until a specific database migration is
confirmed on all environments.

On supplier-facing material grids, quote request and buy-now flows may still
need tighter connection to the real backend in every branch of the UI.

Payment integrations include paths where a mobile money provider is described
as not implemented yet, so users in those flows may hit errors until that work is
completed. Product priority: treat a full Paystack (or other gateway) rollout as
the last major payment fix after other app gaps are closed, so checkout can stay
on the current simulated or partial flows until then.

The suppliers experience on small iPhone-oriented screens still shows “coming
soon” for several sections instead of full feature parity with desktop.


OPERATIONS AND DOCUMENTATION
--------------------------------------------------------------------------------

The site vision worker can run as a manual Python process (see
workers/site-vision/README.md) or as Docker: workers/site-vision/Dockerfile and
docker-compose.yml (compose uses restart: unless-stopped). For Windows, use Task
Scheduler, NSSM, or Docker Desktop restart policy if you need boot-time start
and auto-restart; that host wrapper stays your ops choice (no checked-in MSI).

Single entry point for release checklists (deploy, env, migrations, monitoring
smoke, optional vision worker): docs/OPS_RUNBOOK.md. Long-form detail remains in
the linked docs from that file; there is no separate printed booklet beyond
exporting or printing that runbook plus the sections you need.


REMOTE PTZ AND TWO-WAY AUDIO (MONITORING)
--------------------------------------------------------------------------------

The app can store per-camera flags supports_ptz and supports_two_way_audio and
show a monitoring panel (disabled controls) when those are enabled. Actually
moving the lens or streaming microphone audio from the client browser to the
site requires vendor-specific or protocol work: ONVIF PTZ over a reachable
network path, or a vendor cloud / WebRTC talk-back channel. Camera passwords
and control endpoints must not live in the browser; a Supabase Edge Function,
dedicated gateway, or third-party service would relay signed commands from
entitled users only. Until that gateway exists, the UI is intentional
scaffolding only.


CLOSING SUMMARY
--------------------------------------------------------------------------------

In one sentence, the platform is broad and usable for real pilots; site vision
is a motion-first pipeline with a packaged Docker path for on-site deployment,
but not yet a full custom smart-site CV product; security and notification
layers deserve another pass for strict production; several UI and payment
paths still carry TODO markers; and keeping the vision worker running 24/7 on
site hardware is documented in docs/OPS_RUNBOOK.md and workers/site-vision but
remains your ops responsibility on the physical host (not the hosted web app).
